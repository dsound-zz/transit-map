# NYC Transit Map

Real-time map of NYC subway, Metro-North, and LIRR vehicles. Vehicles are plotted as colored dots on a MapLibre map, polled every 15 seconds from MTA GTFS-RT feeds. Route lines are rendered as a semi-transparent layer beneath the dots.

## Stack

- Next.js 16 App Router, TypeScript strict mode
- react-map-gl v8 (vis.gl/react-maplibre) + MapLibre GL JS v5
- MTA GTFS-RT via `gtfs-realtime-bindings`
- Carto Dark Matter base map (no API key needed)

## File Layout

```
types/transit.ts          — all shared interfaces (StopInfo, ShapePoint, VehicleFeature, RouteLineFeature, …)
lib/constants.ts          — LINE_COLORS, getLineColor(), SUBWAY_FEEDS, COMMUTER_FEEDS
lib/interpolation.ts      — haversine, interpolateAlongPolyline, snapToStop, estimateProgress
scripts/generate-gtfs-lookups.ts  — one-time static data generator (see below)
app/api/transit/route.ts  — GET: fetches all 11 GTFS-RT feeds in parallel, resolves positions, returns GeoJSON
app/api/routes/route.ts   — GET: serves route-shapes.json as a GeoJSON FeatureCollection (loaded at cold start)
app/components/Map.tsx    — client component: renders route lines then vehicle dots, polls /api/transit every 15s
app/page.tsx              — renders <TransitMap />
data/                     — gitignored; populated by the generate script
```

## First-Time Setup

The API routes depend on static JSON files in `data/`. Generate them once (and re-run weekly or after MTA schedule changes):

```bash
MTA_API_KEY=your_key npx tsx scripts/generate-gtfs-lookups.ts
```

Outputs: `stops.json`, `shapes.json`, `trip-shapes.json`, `stop-sequences.json`, `route-shapes.json`.

Set `MTA_API_KEY` in `.env.local` before running `npm run dev`.

## Position Resolution Logic (`app/api/transit/route.ts`)

Three cases handled in order:
1. **GPS coordinates present** (commuter rail) → use directly
2. **STOPPED_AT / INCOMING_AT** → snap to stop coords via `snapToStop`
3. **IN_TRANSIT_TO** → interpolate along shape polyline at 50% (TODO: use TripUpdate timing for precise progress)

## Route Lines (`data/route-shapes.json`)

Generated by selecting, for each `route_id + direction_id` pair, the shape with the most points. More points = full-length variant, not a short-turn stub. No shape merging — that causes duplicate overlapping segments.

Rendered as a `line` layer (opacity 0.6, width 3px) placed before the vehicle `circle` layer in JSX so dots always appear on top.

## Known Issues

- **7-train feed** (`nyct%2Fgtfs-7`) consistently fails to decode — the MTA feed contains proto2 `end-group` markers (wire type 4) that `gtfs-realtime-bindings` doesn't support. The feed is skipped gracefully; 7-train vehicles don't appear.
- **MTA content-type**: all feeds return `Content-Type: text/plain` despite being binary protobuf. Do not filter on content-type.
- **Interpolation is approximate**: IN_TRANSIT_TO uses a fixed 50% fraction. Cross-referencing TripUpdate entities from the same feed would give real progress.
- **NJ Transit**: different API/auth scheme, not yet integrated.
